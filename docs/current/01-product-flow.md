## Purpose

이 앱은 학습자가 막연한 학습 주제나 구체적인 캡스톤 문제를 입력했을 때, 최종 수행 목표를 중심으로 학습 범위를 구성하고, 각 지식 단위의 이해 여부를 생산형 테스트로 검증하도록 돕는다.

핵심은 “무엇을 공부할까?”가 아니라 “무엇을 수행할 수 있게 될까?”에서 출발하는 것이다.

---

## Service Logic

```text
[1] 입력
    ├─ 학습 내용 입력
    └─ 캡스톤 문제/프로젝트 입력

[2] 캡스톤 확정
    ├─ 학습 내용이면 LLM이 canonical 문제/프로젝트 제안
    └─ 캡스톤이면 이진 성공 기준을 정제

[3] DAG 초안 생성
    └─ 캡스톤에서 역추적해 지식 노드와 선수관계 생성

[4] 노드별 지식 이해 테스트 생성
    └─ 각 지식 노드의 baseline 학습 목표로 저장

[5] 학습 진행
    ├─ 선수관계상 학습 가능한 노드 선택
    ├─ 자료 학습, 메모, LLM 질의응답
    └─ 지식 이해 테스트 통과 시 노드 완료

[6] DAG 반복 수정
    ├─ 노드 분해
    ├─ 노드 병합
    ├─ 노드 제외
    ├─ 선수관계 수정
    └─ 전체 리팩토링

[7] 캡스톤 도전
    └─ 모든 필수 선수 노드 완료 후 캡스톤 수행

[8] 학습 성취 로그 출력
    └─ 목표, DAG 변화, 테스트 기록, 오개념 변화, 캡스톤 결과를 문서화
```

## Input Modes

| 입력 모드 | 설명 | 다음 단계 |
|---|---|---|
| 학습 내용 | “복소해석학을 공부하고 싶다”처럼 분야나 주제를 입력 | LLM이 canonical 캡스톤 2~3개 제안 |
| 캡스톤 | “유수 정리로 특정 이상적분을 계산하고 싶다”처럼 수행 목표를 입력 | LLM이 이진 성공 기준을 정제 |

학습자가 학습 내용만 입력한 경우 앱은 바로 DAG를 만들지 않는다. 먼저 해당 분야의 핵심을 대표하는 캡스톤 후보를 제안하고, 학습자가 선택하거나 조정한 뒤 DAG를 만든다.

---

## Capstone Rules

캡스톤은 다음 조건을 만족해야 한다.

1. 성공/실패를 이진적으로 판정할 수 있어야 한다.
2. 계산, 증명, 분석, 구현, 설계, 프로젝트 같은 구체적 수행 형식이어야 한다.
3. “얼마나 잘했는가”는 캡스톤 판정 기준이 아니라 회고나 만족도 필드로 남긴다.
4. 캡스톤은 DAG의 리프 노드이며, 모든 필요한 지식 노드를 선수로 가진다.

---

## DAG Draft Flow

캡스톤이 확정되면 LLM은 다음 질문을 재귀적으로 수행한다.

```text
이 캡스톤을 달성하려면 무엇을 알아야 하는가?
```

도출된 지식 노드는 선수관계 간선으로 연결된다. 초안 DAG는 완성품이 아니라 학습 중 계속 편집되는 살아있는 문서다.

---

## Node Learning Flow

1. 앱은 `getReadyNodes()`로 현재 학습 가능한 노드를 계산한다.
2. 학습자는 노드를 선택해 학습을 시작한다.
3. 노드의 TestGoal을 확인한다.
4. 필요한 공부, 메모, LLM 질의응답을 진행한다.
5. 지식 이해 테스트를 본다.
6. TestGoal의 완료 기준을 만족하면 노드를 완료 처리한다.

---

## DAG Revision Flow

학습 중 언제든 DAG를 수정할 수 있다.

| 상황 | 처리 |
|---|---|
| 노드가 너무 큼 | 해당 노드를 하위 노드들로 분해 |
| 노드가 중복됨 | 노드 병합 |
| 이미 아는 내용 | 제외 처리 |
| 선수관계가 이상함 | edge 수정 |
| 전체 구조가 지저분함 | DAG 전반 리팩토링 |
| 새 캡스톤 추가 | 새 DAG 초안 생성 후 기존 DAG와 통합 리팩토링 |

LLM은 변경을 직접 적용하지 않고 `GraphPatch` 후보를 반환한다. 사용자가 Patch Preview에서 승인하면 앱이 검증 후 적용한다.

## Capstone Attempt Flow

모든 필수 지식 노드가 완료되면 active capstone은 `ready` 상태가 된다.

캡스톤 수행 후 판정은 이진 기준으로 한다.

```text
성공 기준을 만족했다 → achieved
성공 기준을 만족하지 못했다 → failed 또는 retry_needed
```

캡스톤 실패 시 학습자는 다시 DAG를 수정하거나 부족한 노드를 추가할 수 있다.

---

## Achievement Log Flow

캡스톤 달성 후 앱은 학습 성취 로그를 생성한다.

포함 내용:

- 원래 입력
- 확정된 캡스톤과 성공 기준
- 최종 DAG
- DAG 주요 변경 이력
- 완료한 지식 노드 목록
- 노드별 지식 이해 테스트 결과
- 발견된 오개념과 해소 과정
- 캡스톤 시도 결과
- 다음 캡스톤 추천 또는 후속 학습 제안
